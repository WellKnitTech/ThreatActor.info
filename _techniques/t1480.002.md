## Description

Adversaries may constrain execution or actions based on the presence of a mutex associated with malware. A mutex is a locking mechanism used to synchronize access to a resource. Only one thread or process can acquire a mutex at a given time. [Microsoft Mutexes](https://learn.microsoft.com/en-us/dotnet/standard/threading/mutexes)

While local mutexes only exist within a given process, allowing multiple threads to synchronize access to a resource, system mutexes can be used to synchronize the activities of multiple processes. [Microsoft Mutexes](https://learn.microsoft.com/en-us/dotnet/standard/threading/mutexes) By creating a unique system mutex associated with a particular malware, adversaries can verify whether or not a system has already been compromised. [Sans Mutexes 2012](https://www.sans.org/blog/looking-at-mutex-objects-for-malware-discovery-indicators-of-compromise/)

In Linux environments, malware may instead attempt to acquire a lock on a mutex file. If the malware is able to acquire the lock, it continues to execute; if it fails, it exits to avoid creating a second instance of itself. [Intezer RedXOR 2021](https://intezer.com/blog/malware-analysis/new-linux-backdoor-redxor-likely-operated-by-chinese-nation-state-actor/) [Deep Instinct BPFDoor 2023](https://www.deepinstinct.com/blog/bpfdoor-malware-evolves-stealthy-sniffing-backdoor-ups-its-game)

Mutex names may be hard-coded or dynamically generated using a predictable algorithm. [ICS Mutexes 2015](https://isc.sans.edu/diary/How+Malware+Generates+Mutex+Names+to+Evade+Detection/19429/)
